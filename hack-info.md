# Cursor Guild Hackathon Overview

Source: extracted from [Cursor Guild: Bounty Board.html](/Users/admin/projects/hachiko/Cursor%20Guild:%20Bounty%20Board.html) and the rendered text captured in [chatlog.md](/Users/admin/projects/hachiko/chatlog.md).

## Mission

**Build Eric's Software Factory**

Teams are expected to take ideas Eric has talked about and turn them into real, working tools or systems.

## Core Rule

Pick **one** main track and build clearly for that track.

- Each team enters exactly one main track.
- Side quests are optional bonus categories on top of the main entry.
- If a project overlaps multiple tracks, judges will score it on the track they think fits best.

## Scoring

Final score = **main track score** + **judge bonus bucket**

### Main Track Score

Maximum: **100 points**

- Concrete Workflow Value: 30 pts
- Track Fit: 25 pts
- Reliability and Verification: 20 pts
- Technical Execution: 15 pts
- Demo Clarity: 10 pts

### Side Quest Bonus

Maximum: **+30 points total**

- Bonus points come from a single shared judge bonus bucket.
- A team can earn the full bonus from one side quest or split it across several.
- Example splits: `30`, `20 + 10`, `15 + 10 + 5`

## Main Tracks

### 1. Always-On Agents

**Theme:** Live triggers + always-on actions

**Description:** Webhook- and schedule-driven agents that stay useful after the demo.

**Build this**

- Automations triggered by outside events
- Scheduled jobs
- Webhook-driven flows
- Agents triggered by PRs, incidents, or messages

**Not this**

- One-off demos that only run because a person manually starts them

### 2. Review + QA

**Theme:** Checkpoints + quality gates

**Description:** Reliability systems, verifiers, incident responders, and evidence-driven debugging.

**Build this**

- Testing systems
- Verification loops
- Debugging tools
- Review workflows
- Reliability systems that inspect software quality

**Not this**

- General orchestration platforms unless quality control is the core product

### 3. Agent Runtime Tools

**Theme:** Skills + tools + decision layers

**Description:** Skills, MCP tools, model routing, and decision systems that make agents more capable.

**Build this**

- Tools that improve how agents choose models
- Tools that improve MCP usage
- Skill invocation systems
- Decision and reasoning layers for agents

**Not this**

- Full business workflows unless the runtime primitive itself is the main contribution

### 4. Software Factory

**Theme:** Pipelines + fleets + continuous improvement

**Description:** Systems that continuously build, test, coordinate, and improve codebases.

**Build this**

- Multi-step systems
- Several agents or stages working together
- Pipelines that plan, build, verify, and improve software continuously

**Not this**

- A single automation
- A single verifier

## Practical Reading

The hackathon is optimized for:

- Real workflow value
- Strong fit to one track
- Reliability and verification
- Clear scope
- Useful systems that keep working beyond the demo

The most important strategic mistake to avoid is building something broad enough to look like it belongs to multiple tracks.
